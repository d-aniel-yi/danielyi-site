**Section 5 — SAM/ML Pipeline** (id: "sam-ml-pipeline")

Content narrative:
- Opening paragraph: Mobi uses Meta's Segment Anything Model (SAM) for AI-powered image segmentation. Running inference on GPU is expensive — the architecture separates embedding generation from prediction, caching embeddings in Redis so repeated interactions on the same image do not recompute from scratch.
- Second paragraph: When a user uploads an image, a Celery GPU worker loads SAM using `sam_model_registry`, a registry pattern that maps model variant names (e.g., "vit_h") to constructor functions. The model loads once per worker and is cached globally — subsequent calls return the cached predictor.
- Code block 1 (model loading from gpu_sam_tasks.py lines 43-74): Show `get_sam_predictor_on_gpu()` function with the global cache check, `sam_model_registry[sam_model_type](checkpoint=...)`, `.to(device="cuda")`, and `SamPredictor(...)`. Use purple for `def`/`return`/`if`/`global`, yellow for `get_sam_predictor_on_gpu`, green for `"cuda"` and string values, gray-500 for comments.
- Third paragraph: After the model is ready, a second task generates image embeddings. `predictor.set_image()` converts the image into a high-dimensional feature representation. These features (`predictor.features`) are serialized and written to Redis with a 1-hour TTL. The main service polls Redis until embeddings appear before dispatching the prediction task.
- Code block 2 (embedding generation from gpu_sam_tasks.py lines 81-144): Show `generate_embedding_gpu_task` with the `@celery_app.task` decorator, `cv2.cvtColor` call, `predictor.set_image(image_rgb)`, `predictor.features`, `embedding_data` dict construction, and `sam_service.cache_image_embedding(...)` call. Use blue-400 for the decorator, yellow for the function name, purple for `def`/`return`, green for string literals.



**Section 6 — Real-time Notifications** (id: "websocket-notifications")

Content narrative:
- Opening paragraph: Rather than polling the server for task completion, Mobi uses WebSocket connections to push notifications to the browser the moment a background job finishes. The scope is intentionally narrow: the WebSocket channel carries notification events only, not live data streams.
- Second paragraph: `ConnectionManager` maintains a registry of active WebSocket connections keyed by user ID. A `Dict[str, List[WebSocket]]` structure allows a single user to have multiple simultaneous connections (e.g., multiple browser tabs) — each tab receives the notification. An `asyncio.Lock()` prevents concurrent modification of the registry.
- Code block 1 (ConnectionManager from connection_manager.py lines 5-34): Show the class definition, `__init__` with `_connections` dict and `_lock`, the `connect` method with `await websocket.accept()` and `setdefault`, and `send_personal` with the lock, iteration, and `await ws.send_json(data)`. Use yellow for class and method names, purple for `class`/`def`/`async`/`await`/`return`, blue-400 for type annotations, green for string literals.
- Third paragraph: When a Celery task completes, `NotificationService.create_and_dispatch()` persists a `Notification` row to PostgreSQL first, then calls `manager.send_personal()` to fan out to all of the user's connected clients. The persistence-first pattern ensures notifications survive page refreshes — the frontend can re-fetch missed notifications on reconnect.
- Code block 2 (notification_service.py lines 12-20): Show `create_and_dispatch` with `crud_notification.create_notification(...)` call and `await manager.send_personal(notif.user_id, {...})` with the JSON payload fields (id, message, severity, type, created_at). **Section 7 — Workspace UI** (id: "ui-design")

Content narrative:
- Opening paragraph: The Mobi frontend is a multi-panel workspace where every pane can be dragged, resized, and rearranged. Rather than building custom drag handlers, the app delegates this entirely to `react-mosaic-component` — a tiling window manager for React.
- Second paragraph: `LayoutManager` is a thin controlled wrapper. It receives `layout` (the current tile tree), `renderTile` (how to render each view), and `onChange` (callback when the user repositions a panel). The underlying `Mosaic` component handles all hit-testing, pointer events, and animation. Panels can be resized down to 1% of available space via `minimumPaneSizePercentage`.

- Third paragraph: The bottom toolbar uses Material-UI icons — pre-built accessible SVG components imported by name. Each panel type maps to a semantic icon: a ruler for the measurement tool, a clipboard for tasks, a chart for results. No hand-drawn SVGs or custom paths are needed.
